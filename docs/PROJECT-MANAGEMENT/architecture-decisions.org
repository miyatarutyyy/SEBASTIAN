* ADR-0001: Steward のレイヤリングと委託モデル

** Status

Accepted (2026-07-24)

** Context

- 現状の実装では、Steward が起床シーケンスを直接所有している。
  - =StewardState= に =wakeStartedAt= / =wakeCompletedAt= という起床ドメイン固有の
    フィールドが埋まっている（=apps/web/features/steward/steward.ts=）。
  - 遷移関数も =startWaking= / =completeWaking= / =resetToSleeping= と、すべて起床専用。
  - 保存キーも =sebastian:steward:state= という単一の塊。
  - つまり「Steward ＝ 起床管理システム ＝ 起床シーケンス」が一枚に潰れている。
- 一方で Steward は、将来的に多数の小アプリ（スケジュール管理・タスク管理・食事管理・
  運動管理・継続管理・日記管理など）を束ねる上位オーケストレーターとして構想されている。
- サイトマップを描く議論の中で、「Steward が起床シーケンスを直接管理している」ように
  見えることが判明し、抽象レイヤーが一段足りないと結論した。

** Decision

1. 抽象を3層に分ける。
   - =Steward=（執事長 / オーケストレーター）: 子分システムを登録・起動・調停する。
     ドメインの中身は知らない。自分では手を動かさず委託する。
   - =○○管理システム=（子分 / ドメイン）: 1つの生活領域の責務を丸ごと持つ。
     内部に状態・ルール・保存・シーケンス・UI を持つ。例: 起床管理システム。
   - =シーケンス=（手続き / 状態遷移）: 管理システム内の具体的な流れ。
     起床シーケンス = =sleeping -> waking -> awake=。

2. 生活状態 =sleeping / waking / awake= の所有は「起床管理システム」に閉じる（案 a）。
   - 起床管理システムが唯一の書き手（真実の源）。
   - Steward や他のドメインは参照のみ（single writer, many readers）。

3. Steward の役割は「委託 ＋ 交通整理」であり、ドメインの仕事そのものはしない。
   - 現在アクティブな子分を前面に出す調停を担う（提示の交通整理）。

4. 執事長と子分の契約は「申告制」とする（案 Q）。
   - 子分が「提示要求」を申告する（例: =none | normal | exclusive= ＋ 優先度）。
   - Steward はドメインの中身を知らず、優先度だけで前面を調停する。
   - 「寝ている間は他の小アプリを出さない」は、起床管理システムが waking/sleeping の間
     =exclusive= を申告し、Steward が優先度で裁くことで実現する。

5. スコープは段階的に進める。
   - 今は Y（申告 -> 調停による提示の交通整理）まで実装する。
   - Z（ドメイン間連携。例: 起床完了 -> 今日のスケジュール提示）は、まだ実装しない。
     Steward の中核を後から書き換えずに Z を足せるよう、イベントの受け渡し口という
     「縫い代」だけを残す。

** Considered alternatives（見送った案）

*** 案 b: 生活状態を Steward が所有する

- 内容: =sleeping / waking / awake= を Steward が全体の生活フェーズとして所有する。
- 見送った理由:
  - 現状のレイヤー崩壊（Steward が起床の中身を知っている状態）を固定化してしまう。
  - 「寝ている間は他を出さない」体験は、所有を移さずとも参照だけで実現できる。
  - 将来どうしても全体フェーズが必要になっても、所有を動かさずに Steward が参照する
    形へ育てられるため、先に (b) を選ぶ必要がない。

*** 案 P: Steward がドメインのルールを持って裁く

- 内容: 「waking の間は起床アプリだけ」といったルールを、Steward が各ドメインの事情を
  知った上で持ち、直接裁く。
- 見送った理由:
  - Steward がドメイン知識を抱え込み、子分が増えるたびに Steward を書き換える必要が出る。
  - それは今回まさに解消しようとしたレイヤー崩壊の再発であり、「執事長は委託する」という
    設計思想に反する。
  - 代わりに案 Q（申告制）を採用することで、Steward はドメイン非依存を保てる。

** Consequences

- 将来のリファクタとして、=StewardState= から起床固有フィールドを剥がし、起床管理システム側へ
  移す必要がある（現状のコードはまだこの決定に対応していない）。
- 子分システムが共通で満たすインターフェースを定義する必要がある。少なくとも:
  - =id=（例: =wake= / =schedule=）
  - 提示要求の申告（=none | normal | exclusive= ＋ 優先度）
  - 発するイベント / 受けるコマンド
  - 状態・保存・UI は各子分が自前で持つ（Steward は触らない）
- Steward が全子分へ提供するもの:
  - 共有コンテキスト（現在時刻などの時計）
  - イベントの受け渡し口（Z の縫い代）
  - 優先度による前面の調停
- 保存は将来的に =sebastian:<module>:state= のようにドメインごとへ分割する候補となる。

** Open questions（未決の論点）

- Z（ドメイン間連携）を入れるときの「連携の知識」の置き場所。
  - オーケストレーション: 執事長が宣言的な連携シナリオ（データ）として把握する。
  - コレオグラフィ: 子分同士が互いのイベントに反応して自律的に動く。
  - いずれを採る場合も、Steward のコードはイベント名 / コマンド名だけで繋ぎ、
    ドメインの内部フィールドは読まないことを前提とする。
- 子分共通インターフェースの具体的な形。
- 保存の分割方針とマイグレーション。
